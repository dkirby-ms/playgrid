### Recommendation

Implement server-side bot player with greedy heuristic AI (start with Medium difficulty).

### Complexity
- **Estimate:** Small (~250 LOC)
- **Scope:** Server-side only (new file: CpuOpponent.ts, ~15 lines lobby/room changes)
- **Risk:** Low (reuses existing plugin system, no schema changes)

### Architecture
1. **Bot Creation:** Lobby creates CPU opponent when single-player game is spawned
2. **Bot Registration:** CPU is a PlayerInfo with synthetic sessionId ("cpu-opponent")
3. **Move Selection:** On CPU turn, call selectCpuMove() and apply automatically (no network)
4. **Timing:** 200–500ms delay for UI feedback

### AI Strategy
**Greedy Heuristic (Recommended)**
- Prefer captures (highest priority)
- Prefer promotions (advance to king row)
- Prefer advancement (move toward opponent)
- Random tiebreaker (unpredictability)

Alternative: Minimax with α-β pruning (deferred to Phase 2 if needed).

### MVP
- Greedy heuristic only (not random, not Minimax)
- Medium difficulty only
- Checkers only
- No persistent profiles

### Nice to Have (Phase 2)
- Easy/hard difficulty modes
- Minimax for stronger AI
- Win/loss stats

### Files to Create/Modify
- **New:** `server/src/games/checkers/CpuOpponent.ts` (~80 lines)
- **Modify:** `LobbyRoom.ts` (~15 lines), `BaseGameRoom.ts` (~20 lines)
- **Test:** `checkersPlugin.cpu.test.ts`, `lobby-cpu.test.ts` (~150 lines total)

### Why Server-Side Bot
- Client-side AI can be cheated (not viable)
- Separate AI service overkill for Checkers (adds latency, complexity)
- Server-side is straightforward: bot is just a Player with synthetic sessionId

### Why Greedy Heuristic
- Fast: O(m log m) where m ~= 12 legal moves
- Playable: captures, promotes, feels real
- Clean: evaluation function isolated, testable
- Scalable: can add Minimax later without rewriting core
