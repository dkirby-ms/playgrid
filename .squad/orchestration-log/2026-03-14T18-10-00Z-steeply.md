**Test Coverage (`e2e/checkers.spec.ts`):**
1. Checkers game initializes with correct board state
2. Move validation and execution (31-move deterministic sequence)
3. Piece promotion on reaching back rank
4. King movement rules (backward allowed)
5. Win condition detection (no valid moves for opponent)
6. Win/loss message display

**Deterministic 31-Move Sequence:**
- Covers all major rule paths: normal moves, promotion, king movement, final win
- Reproducible across environments (no timing, no visual assertions, no coordinate-based clicks)

**Result:**  
3 files changed, +524 lines

**Key Learnings:**
- Canvas E2E is flaky across environments; avoid coordinate-based clicks
- Grey box pattern: browser harness for deterministic state changes, DOM assertions for player-facing messaging
- State synchronization (not pixel output) is what matters for game correctness
- Checkers moves sent through real `gameRoom` object keeps coverage end-to-end without fake test clients
